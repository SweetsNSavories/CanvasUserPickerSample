Building an Outlook-style People Picker in a Canvas App — live, agent-driven

A field report on driving Power Apps Studio end-to-end through a browser agent, the Power Fx gotchas we hit, and the small workarounds that turned a flaky build into a reliable one.

May 28, 2026  ·  Environment: SeaCass  ·  App: CanvasUserPickerSample

The goal

We wanted a low-code, Outlook / InfoPath style people picker inside a Power Apps Canvas app:

  • Type a partial name → live tenant-directory suggestions.
  • Multi-select with checkboxes.
  • Picks survive when the user starts a new search — the classic InfoPath "fix-up" behavior where one wrong name doesn't force re-typing the rest.

Canvas comboboxes can do this, but only when the underlying state is wired correctly. The interesting thing about this build wasn't the Power Fx — it was that we drove Power Apps Studio itself through a browser agent: every click, every formula edit, every save was a tool call.

TL;DR

Five properties on a single Combobox + a one-line OnStart get you the full Outlook-style picker. The real cost was discovering which surfaces of the Power Apps Studio UI are reliable to automate.

Setup

The remaining work was to actually build the app in the maker portal, which the agent did live in the VS Code integrated browser.

Lesson 1 — The main formula bar is a trap. Use the Advanced tab.

Power Apps Studio shows formulas in two places:

  1. The big formula bar at the top of the page — a single, long-lived Monaco editor instance.
  2. Per-property inline editors on the right-side Properties → Advanced tab.

Both render identical-looking Monaco controls. They behave very differently under automation:

SurfaceBehavior when we cleared + typed a new formula
Main formula bar (top)Residual tokens leak in. clear: true is unreliable. We ended up with half of the previous formula concatenated to the new one.
Advanced-tab inline editorSpawns a fresh Monaco instance on every property click. clear: true works cleanly. Every typed formula round-tripped without artifacts.

This single discovery is what turned the build from frustrating to mechanical. Once we committed to editing via the Advanced tab, the agent could set five Combobox properties in a row without a single retry.

Committing a formula is just as simple: click any other property button on the same panel. The editor blurs, Power Fx revalidates, the error indicator either clears or appears next to the property name, and you can move on.

Lesson 2 — ShowColumns doesn't take strings anymore

Our first OnChange formula tried to project the Combobox's selected items into a smaller-shape collection: 

// FAILS in modern Power Fx with: "Expected identifier name"
ClearCollect(
    colPeople,
    ShowColumns(Self.SelectedItems, "DisplayName", "Mail", "JobTitle", "UserPrincipalName", "Id")
)

Old reference material (and a lot of Stack Overflow) shows ShowColumns with quoted string column names. Modern Power Fx requires bare identifiers instead: 

ShowColumns(Self.SelectedItems, DisplayName, Mail, JobTitle, UserPrincipalName, Id)

But the better fix was to delete ShowColumns entirely — which leads directly to the next lesson.

Lesson 3 — The schema-inheritance trick for collections

Combobox.DefaultSelectedItems must be type-compatible with Combobox.Items. If Items returns the full SearchUser record shape but colPeople is a stripped-down 5-field record, you get:

Invalid formula. Expected a value compatible with 'Items'.

Rather than enumerate the exact SearchUser shape (which would couple us to whatever fields Microsoft adds tomorrow), we let the connector tell us its own schema by priming the collection with a no-match call and immediately clearing it: 

// App.OnStart
ClearCollect(
    colPeople,
    Office365Users.SearchUser({ searchTerm: "zzznotfound_init_xyz" })
);
Clear(colPeople);
Set(gblRecipientsCsv, "");

Now colPeople has the exact shape Items returns. DefaultSelectedItems = colPeople binds without complaint, and OnChange can write Self.SelectedItems back into colPeople directly — no ShowColumns, no field-by-field rebuild.

Why this matters

Connector return-shapes change. Hand-typed schemas drift. Letting the data source seed your collection's shape removes a whole class of "looks fine, breaks in two months" bugs.

Lesson 4 — The DefaultSelectedItems ↔ OnChange round-trip

This is the actual mechanism that makes "picks survive a new search" work in a single Combobox:

  1. User picks Alice. Combobox writes Self.SelectedItems = [Alice].
  2. OnChange fires → ClearCollect(colPeople, Self.SelectedItems). Now colPeople = [Alice].
  3. User starts typing "bob". Items re-runs with the new search text — the dropdown now shows Bobs.
  4. Internally Power Apps re-evaluates the Combobox. DefaultSelectedItems = colPeople means Alice stays selected even though she's not in the new Items result set.
  5. User picks Bob. OnChange fires again → colPeople = [Alice, Bob]. Done.

Without the collection on both sides of that loop, step 4 would silently drop Alice the instant the dropdown refreshed. This is the same dance Outlook does internally; we just expressed it in five lines of Power Fx.

Final formulas

WhereFormula
App.OnStart 
ClearCollect(colPeople,
  Office365Users.SearchUser({searchTerm: "zzznotfound_init_xyz"}));
Clear(colPeople);
Set(gblRecipientsCsv, "")
Combobox1.Items Office365Users.SearchUser({ searchTerm: Self.SearchText })
Combobox1.ItemDisplayText ThisItem.DisplayName
Combobox1.InputTextPlaceholder "Type a name or email..."
Combobox1.DefaultSelectedItems colPeople
Combobox1.OnChange ClearCollect(colPeople, Self.SelectedItems)
Combobox toggles SelectMultiple = true, IsSearchable = true

After editing OnStart, you have to run it once explicitly: Tree view → App → More options → Run OnStart. Otherwise colPeople doesn't exist yet and DefaultSelectedItems can't bind.

Validation in Preview

Press F5. Type a few characters; tenant-directory suggestions appear. Pick one — a "Clear selection" button shows up next to the chevron, confirming the pick is committed. Clear the search text, type something completely different, pick a second person. Exit preview and look at the Combobox state in the edit canvas: it displays the literal string "alyousse admin, Praveen Thonda", proving DefaultSelectedItems correctly rehydrates from colPeople across re-renders.

That's the whole picker. No Gallery, no fix-up button, no reorder logic — just one Combobox with the right wiring.

We built the minimal version live to validate the core mechanism. The Gallery + reorder extensions are pure Power Fx — they don't require any new automation lessons, so they're left as a copy-paste exercise from the README.

Meta: what makes Power Apps Studio agentable

Three things had to be true for the agent to drive the maker portal reliably:

  • Accessible refs. The studio renders inside an iframe; every interactive element exposes an ARIA label. The agent's browser tool surfaces these as stable refNNN identifiers that survive across read_page snapshots in the same DOM cycle.
  • Per-property editors. See Lesson 1. Without the Advanced tab's fresh-Monaco-per-click pattern, the entire workflow would have devolved into screenshot-driven recovery.
  • A single page, never a new tab. All studio interaction stayed inside one shared browser page, so the agent could chain read_page → click_element → type_in_page without context loss.

The remaining flake is the property-panel re-render: every commit invalidates the previous ref list, so the next action always starts with a fresh read_page. That's tolerable. What isn't tolerable — and what almost killed this build — was the main formula bar's residual-state problem. Avoiding it cost nothing once we knew about it.
